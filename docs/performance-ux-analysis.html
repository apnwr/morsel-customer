Top 5 Highest-ROI Fixes

  • 1
    Cache sessionUserId in useRef

    Eliminates 6+ synchronous localStorage reads per cart operation — main thread unblocked on every mutation.

  • 2
    Single Firebase participant subscription in SessionContext

    Eliminates duplicate Firebase listeners and duplicate API polling from ParticipantsList. Cuts session API calls by ~60%.

  • 3
    Debounce menu search 300ms

    Prevents synchronous filtering of 100+ items on every character typed. Makes search feel instant rather than janky.

  • 4
    Error boundaries on cart and menu pages

    Any thrown error currently crashes the full page, forcing a hard reload. Error boundary gives users a recovery path.

  • 5
    Skeleton loaders on cart page and order tab switch

    Biggest visible UX gap. Plain <p>Loading...</p> text on the cart page; blank state on tab switch.

Critical — Fix Immediately

CRITICAL MEMORY LEAK
Firebase unsubscribe race condition
src/lib/firebase/realtime.service.ts:402–514

unsubscribeRef is set inside an async auth promise chain. If the component unmounts before auth resolves, the Realtime DB listener stays active permanently. Rapid mount/unmount cycles (e.g. tab switching) accumulate listeners that are never cleaned up.

Memory grows over the lifetime of the session. Stale listeners receive and process data after the component is gone, causing phantom state updates and wasted network.

Fix Move auth initialization outside the subscription callback, or track the pending auth promise in a ref and cancel it on unmount before it resolves.
CRITICAL PERF
6+ synchronous localStorage reads per cart operation
src/contexts/CartContext.tsx:171, 325, 433, 496, 528, 741

getFromStorage('morsel_session_user_id') is called inside every cart mutation — addItem, removeItem, syncCartFromAPI, confirmOrder, and more. Each call is a synchronous main-thread operation.

Blocks the JS thread 6+ times per cart interaction. On low-end Android devices (common for restaurant patrons), this causes visible jank on quantity changes and add-to-cart taps.

Fix Read sessionUserId once during CartContext initialization. Cache in a useRef. Update the ref only when the session changes. Zero reads inside cart operations.
CRITICAL API DUPLICATION
Participants data subscribed in two places simultaneously
src/contexts/SessionContext.tsx:412–447 & src/components/session/ParticipantsList.tsx:229–241

Both SessionContext and ParticipantsList independently subscribe to participants updates — creating duplicate Firebase subscriptions and duplicate API polling every 10 seconds for the same data.

Double the API load on the session endpoint. Two separate processing pipelines that can race. Participants data can momentarily differ between the two sources causing flicker.

Fix Single subscription source in SessionContext only. ParticipantsList reads participants state from SessionContext via useSession(). Remove Firebase subscription and polling from ParticipantsList entirely.
CRITICAL WATERFALL
confirmOrder triggers sequential API waterfall
src/contexts/CartContext.tsx:819–829

confirmOrder → await response → refreshSessionData → cart update. Three sequential async operations where the session refresh could run in parallel with the order confirmation response processing.

1–2 extra seconds before the user sees post-order UI. The "Placing order…" state lasts longer than necessary, making the app feel slow at the most important moment.

Fix Use Promise.all() to run session refresh in parallel with order response processing where data dependencies allow. Transition UI optimistically on confirm tap, rollback on failure.

High — Fix Within Sprint

HIGH POLLING
Two independent polling loops for same session endpoint
src/contexts/SessionContext.tsx:362–376 & src/components/session/ParticipantsList.tsx:270–281

Both contexts poll GET /ordering-session/session/:id every 10 seconds independently. No coordination, no backoff on failure — if the API is slow, both fire simultaneously.

Double the server load from every active customer session. On a busy service period, this compounds across hundreds of sessions.

Fix One polling source in SessionContext. Add exponential backoff: on failure, double the interval up to a max (e.g. 60s). ParticipantsList consumes from context — no polling of its own.
HIGH RE-RENDER
SessionContext value breaks memoization on every render
src/contexts/SessionContext.tsx:466–482

Context value is wrapped in useMemo, but many callbacks inside are created inline without useCallback. Every render of SessionProvider creates new function references, breaking the memo and causing all 20+ consuming components to re-render.

Cascading re-renders through the entire component tree on every session state change. This includes ParticipantsList, Header, CartItem, OrderTabs — all re-render unnecessarily.

Fix Wrap each callback with useCallback independently before composing them into the context value object. Separate mutable state from stable callbacks into different context providers if needed.
HIGH PERF
Menu search filters 100+ items synchronously on every keystroke
src/app/menu/page.tsx:498–524

hasAnyFilteredItems iterates through all menus → all categories → all items and runs mapApiItemToMenuItem() on every character typed in the search box. No debounce, no deferred execution.

On a restaurant with 50+ items, typing quickly causes visible input lag. The search field feels sticky — every character is delayed by the compute time of the filter pass.

Fix Debounce the search query state update by 300ms: const debouncedQuery = useDebounce(searchQuery, 300). Then use useDeferredValue(debouncedQuery) for the filtered list computation to keep the input responsive.
HIGH UX
No error boundaries — any error crashes the full page
src/app/cart/page.tsx & src/app/menu/page.tsx

Neither the cart page nor the menu page has an error boundary. A thrown error anywhere in the component tree (network failure, null deref, API shape mismatch) unmounts the entire page with a blank screen.

Users are stranded with a blank screen and no recovery option. In a restaurant setting where the customer is actively ordering, this is a critical UX failure.

Fix Add a React error boundary around each page's main content area with a "Something went wrong — tap to retry" fallback UI. Next.js also supports error.tsx per route segment as a simpler alternative.
HIGH UX
Plain text "Loading..." on cart page — no skeleton
src/app/cart/page.tsx:70

While cart data fetches, the page shows a bare <p>Loading...</p> string with no visual structure — no skeleton, no spinner, no layout hints.

The page looks broken during load. Users don't know if the app is doing anything. Content jump when data arrives is jarring — no layout stability.

Fix Replace with skeleton components matching the actual layout: cart item skeletons (image box + 2 lines of text), bill section skeleton (3 rows), CTA button skeleton. Use pulsing animation via Tailwind animate-pulse.
HIGH UX
Blank state on order tab switch — no loading indicator
src/components/order/PostOrderView.tsx

When a user taps a different participant's order tab, the component fetches that order's data but shows nothing while loading — momentary blank or stale content.

Tab switch feels broken — user taps, nothing happens visually, then content appears 500ms–1s later. Common pattern in group ordering where users browse others' tabs.

Fix Show order item skeletons immediately on tab switch while the data fetches. Keep the previous tab's total visible in the tab bar (already works) but add a shimmer to the content area.

Medium — Fix When Refactoring

MEDIUM PERF
Dietary type recalculated in 3 separate components
src/components/cart/CartItem.tsx:23–36 & src/components/menu/MenuItem.tsx:162–175 & src/components/order/PostOrderView.tsx:25–41

The same allergen array processing and dietary type derivation logic is duplicated across three components — all computing the same string operations on every render.

Fix Single getDietaryType(item) utility in src/lib/utils.ts. Memoize result per item ID using a Map cache. All three components import and call it identically.
MEDIUM MEMORY
Module-level session cache in ParticipantsList never clears
src/components/session/ParticipantsList.tsx:19–30

A global sessionCache object is defined at module level and persists across component mounts and unmounts. Different sessions can receive stale cached data from a previous session.

Fix Move cache into component state or useRef so it resets on mount. Add a TTL (e.g. 30s) after which the cache is considered stale and re-fetched.
MEDIUM COMPONENT SIZE
CustomizationModal is 695 lines — too many responsibilities
src/components/order/CustomizationModal.tsx

A single file handles: state initialization from item data, price calculations, form validation, spice level selector, addon accordion logic, and rendering — 695 lines of mixed concerns.

Fix Extract SpiceLevelSelector component, AddonGroup component, and a useCustomizationState(item) hook for price/validation logic. Modal becomes a thin coordinator.
MEDIUM COMPONENT SIZE
ParticipantsList is 540 lines — mixes data and presentation
src/components/session/ParticipantsList.tsx

Manages Firebase subscriptions, session caching, participant sync, split calculations, and rendering all in one file. Data logic and UI logic are deeply entangled.

Fix Extract all Firebase and API logic into a useParticipantsSync() hook. ParticipantsList becomes a pure presentation component that receives participants as props.
MEDIUM RE-RENDER
Header recalculates participant dot positions on every render
src/components/layout/Header.tsx:160–177

Array.from() and trigonometric calculations for participant avatar dot positioning run on every render, even when participant count hasn't changed.

Fix const dots = useMemo(() => calculateDots(participantCount), [participantCount]) — recalculates only when count changes.
MEDIUM UX
Order confirmation waits for full API round-trip before UI transitions
src/hooks/useCartPageState.ts:192–232

After user taps "Place Order", the app waits 1–2 seconds for the API response before transitioning to post-order view. No intermediate state shown.

Fix Optimistically transition to a "processing" post-order state immediately on tap. Show a subtle spinner in the order summary. Roll back to pre-order view with an error toast if the API fails.
MEDIUM UX
Menu distinguishes "no results" and "API error" with same empty state
src/app/menu/page.tsx:532–540

When the menu API fails, the page shows "No items found" — identical to a genuine empty menu. Users can't tell if their data simply didn't load vs. the menu is truly empty.

Fix Track error state separately from empty state. Show "Couldn't load menu — tap to retry" with a retry button on error. Reserve "No items found" for genuine zero-result states.
